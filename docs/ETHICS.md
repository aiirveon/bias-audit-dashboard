## 1. Why this document exists

AI systems that detect bias can themselves produce biased outcomes. A tool built to protect fairness can cause harm if it misclassifies systematically, if its explanations mislead reviewers, or if it is used to automate decisions that should remain human.

This document defines the ethical principles this product is built on, the constraints those principles impose on the system design, and the ongoing responsibilities of anyone who builds, deploys, or uses it.

Drawing on Cathy O'Neil, Weapons of Math Destruction: a model is dangerous when it is opaque, operates at scale, and causes harm to the people it scores. This product is designed to be the opposite — transparent, bounded in scope, and subordinate to human judgement.

## 2. Core ethical principles

### 2.1 Human agency is non-negotiable

The system never makes a content decision. It provides a signal. The reviewer makes the decision.

Every result card requires an explicit reviewer action: Approve, Flag, or Escalate. There is no auto-approve. There is no auto-remove. The system cannot suppress or publish content on its own.

This is not a UX choice. It is an ethical constraint embedded in the architecture.

### 2.2 Explainability is a right, not a feature

A reviewer who cannot understand why a piece of content was scored high-risk cannot make a defensible decision. Every result includes:

- The specific words that triggered the score (SHAP highlights)
- A plain English explanation generated by the Claude API
- The confidence level of the prediction

### 2.3 The model must not discriminate in its detections

A bias detection tool that flags one category's content at a systematically higher rate than another is itself a source of bias.

Fairness constraint (design target): no bias category may be flagged at more than 2× the rate of any other category in real-prediction outcomes.

**v1 implementation status:** v1 reports a dataset balance ratio — the max/min count of biased-labelled examples per category in the synthetic training CSV. This checks that the dataset was constructed evenly; it does not measure whether the model's predictions on real content are fair. Prediction-level fairness evaluation (per-category false-positive rates, equal opportunity, predictive parity) is planned for v2 using Fairlearn and is not implemented in v1. See `docs/MODEL_DECISIONS.md` for the full gap statement.

### 2.4 Uncertainty must be communicated honestly

The model is not certain. Confidence scores reflect this. The system never presents a score as a fact. It presents it as a probability that requires human interpretation.

### 2.5 Synthetic data is a deliberate and bounded choice

The training dataset is synthetic — generated using the Claude API and reviewed manually. No real user content is stored or processed during training. No individual is profiled or scored.

## 3. What this system must never do

| Constraint | Rationale |
|-----------|-----------|
| Never automatically remove or suppress content | Removal without human review is censorship without accountability |
| Never store full text of real user content in v1 | Privacy — no content processed in the demo is retained |
| Never present a score as a final verdict | The score is a signal. The reviewer makes the verdict. |
| Never expose API keys or model weights publicly | Security — all credentials in environment variables |
| Never claim the model is unbiased | No model is unbiased. The claim itself is a harm. |

## 4. Bias risks in this system

### 4.1 Training data bias

If the synthetic dataset over-represents certain writing styles associated with one group, the model may learn spurious correlations.

Mitigation: dataset generation prompts diversified across regions, registers, and cultural contexts. ~500 examples per category. Reviewed manually before training.

### 4.2 SHAP explanation bias

SHAP highlights can mislead if the model learned spurious correlations rather than genuine bias signals.

Mitigation: SHAP outputs validated on 50 test cases before deployment.

### 4.3 Automation bias

Reviewers may over-trust a high-confidence verdict and stop reading the content critically.

Mitigation: UI shows content snippet first, score second. Reviewer action is always required. The card cannot be dismissed without a deliberate choice.

### 4.4 Confidence calibration

A model can be 95% confident and wrong. Confidence scores reflect model certainty, not ground truth correctness.

Mitigation: calibration curve evaluated during model assessment. Documented in /docs/MODEL_DECISIONS.md.

## 5. Regulatory context

### Online Safety Act 2023 (UK)

Places a duty of care on platforms to protect users from illegal and harmful content. This product supports compliance by providing a structured, auditable review process and an exportable audit log. It does not constitute legal compliance on its own.

### Ofcom

Regulates broadcasters and on-demand services. The audit dashboard and audit log provide aggregate evidence of bias monitoring activity presentable to Ofcom. Full prediction-level fairness metrics (§6) are planned for v2 — v1 reports dataset balance only.

### UK GDPR

In v1, no real personal data is processed. If real content is processed in future versions, a Data Protection Impact Assessment (DPIA) must be completed first.

## 6. Fairness metrics — definitions (v2 design targets, not computed in v1)

**Status: these four metrics are defined here as design targets for v2. None of them are computed in v1.** v1 only computes a dataset balance ratio (max/min biased-example count per category in the synthetic CSV), which is described in `docs/MODEL_DECISIONS.md`. The definitions below are preserved because they represent the intended fairness evaluation design and will become acceptance criteria for v2.

### Demographic Parity

Flag rate per category should be within ±15% of the mean flag rate.

Plain English: "The model flags roughly the same proportion of content in each category."

### Equal Opportunity

True positive rate per category within ±10% of the mean.

Plain English: "When content is actually biased, the model catches it at roughly the same rate regardless of category."

### Predictive Parity

Precision per category within ±10% of the mean.

Plain English: "When the model raises an alert, it is right at roughly the same rate across all categories."

### Individual Fairness

Average score difference for matched content pairs below 0.10.

Plain English: "Similar content is scored similarly regardless of which group is referenced."

## 8. What good looks like

A trust and safety analyst using this tool should be able to say:

> "I understand why this content was flagged. I can see the specific words the model reacted to. I read the explanation. I made my own decision. I logged it. If Ofcom asks, I can show them every step."

That is the standard this product is built to meet.
